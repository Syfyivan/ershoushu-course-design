老师好，我本次课程设计的题目是校园二手书交易平台。这个项目主要解决校园内教材和二手资料流通效率低的问题。传统方式依赖微信群、贴吧或者线下摆摊，信息比较分散，也缺少订单记录和统一管理。所以我设计了一个基于 Spring Boot、MyBatis-Plus、MySQL 和 Vue 的 B/S 架构系统。 这次是中期检查，我主要展示当前已经完成的需求分析、数据库设计、后端接口、前后台页面和核心模块联调情况。最终交易闭环和测试截图会在下一阶段继续完善。

数据库设计

约 60 秒

画面

打开 ershoushujiaoyipingtai/db.sql，定位 CREATE TABLE 列表。

这是数据库初始化脚本，数据库名是 ershoushujiaoyipingtai。当前一共设计了 13 张表。这里的 users 是管理员账号表，yonghu 是普通用户表，token 用于保存登录令牌。 业务核心是 tushu 图书表，它保存图书名称、作者、出版社、类型、库存、价格、图片、上下架状态和逻辑删除状态。交易相关表包括 cart 购物车表、tushu_order 订单表和 address 收货地址表。互动相关表包括 tushu_liuyan 图书留言、tushuqiugou 图书求购、chat 用户反馈和 news 公告。 这部分体现的是数据库课程设计里的实体完整性和参照关系设计，后端接口会围绕这些表完成增删改查和业务状态流转。

后端架构和登录鉴权

约 70 秒

画面

打开 UsersController.java 和 AuthorizationInterceptor.java。

后端采用 Spring Boot，包结构按分层组织。controller 负责接口入口，service 负责业务服务，dao 负责数据库访问，entity 对应数据库表，utils 里放统一返回对象、分页和通用工具。 这里以登录为例，UsersController 的 /users/login 接口会根据用户名查询用户，校验密码后调用 TokenService.generateToken 生成 token，并把 token、角色和用户 id 返回给前端。后续请求会通过拦截器校验 token。标注了 @IgnoreAuth 的登录、注册接口可以匿名访问，其他接口需要登录后才能访问。 这一部分是中期阶段已经完成的系统入口能力，也是后续所有业务模块的权限基础。

约 90 秒

画面

打开 TushuController.java，再切到后台图书管理页面。

操作建议

  1. 在后台登录 admin/admin。
  2. 打开“图书信息管理”。
  3. 展示列表、搜索、新增或编辑弹窗。
  4. 如果数据库已导入，可展示一条示例图书详情。
图书模块是本系统的核心模块。后端对应 TushuController，已经实现了分页查询、详情查询、保存、修改、逻辑删除、批量导入和前台列表接口。 在分页查询中，系统会根据当前角色判断查询范围。如果是普通用户，只查看自己发布或相关的数据；管理员可以查看全部数据。图书删除这里没有直接物理删除，而是把 tushu_delete 逻辑删除字段改为删除状态，这样可以避免误删数据，也方便后续恢复或审计。 页面上可以看到后台有图书信息管理模块，管理员可以查看、添加、修改和删除图书。前台用户也可以在图书列表里浏览图书详情，这是后续购物车和订单流程的入口。

常见追问回答

答辩备用

为什么选二手书交易？

校园内每学期都有大量闲置教材，二手书交易是真实需求，能体现用户、商品、订单、留言、公告、地址等数据库对象之间的关系，适合作为数据库和 Web 综合课程设计。

数据库有哪些完整性设计？

所有表都有主键 id；用户登录名做唯一性检查；订单表关联用户、图书、地址；图书用库存、价格、上下架和逻辑删除字段保证业务状态；token 表记录用户、角色和过期时间。

中期最核心的成果是什么？

不是最终界面，而是需求、数据库、架构和核心接口已经落地，并且 Maven 构建通过。后续主要是业务流程联调和测试材料完善。

你自己负责哪部分？

按实际分工替换。建议说：我主要负责需求整理、后端框架搭建、登录鉴权、图书信息管理、数据库脚本整理和中期交付文档；订单和测试部分会和组员继续联调。
